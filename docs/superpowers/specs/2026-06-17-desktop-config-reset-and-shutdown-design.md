# 桌面模式配置初始化与程序退出设计

## 目标

为 KRL Parser 桌面模式增加两项能力：

1. 用户可以将磁盘中的 `Config.yml` 恢复为 `krl-core/src/main/resources/config.yml` 所提供的内置默认内容，并立即将默认内容用于本次分析。
2. 用户可以从 Web 界面触发 Java 程序优雅退出，释放 Web 服务、分析线程池和日志等资源，无需手动终止进程。

两项能力只允许在 `desktop` 运行模式使用。在 `server` 模式下，前端不展示入口，后端也必须拒绝请求。

## 非目标

- 不增加在线保存任意编辑内容到磁盘的能力。
- 不允许服务器模式远程重置配置或关闭服务。
- 不引入 Actuator shutdown 端点或额外的身份认证体系。
- 不改变现有“编辑框内容仅用于本次分析”的语义。

## 方案选择

采用专用业务接口并在后端强制执行桌面模式检查：

- `POST /api/config/reset` 重置配置。
- `POST /api/runtime/shutdown` 调度应用退出。

不使用 Actuator shutdown，因为它会扩大管理端点的暴露面；不直接在控制器请求线程中调用 `System.exit()`，因为浏览器可能收不到响应，Spring 托管资源也可能来不及正常关闭。

## 后端设计

### 桌面模式访问控制

配置重置与退出接口在执行业务逻辑前检查 `KrlRuntimeProperties`。只有 `RuntimeMode.DESKTOP` 可以继续；`RuntimeMode.SERVER` 返回 HTTP 403 和明确的错误信息。后端检查是安全边界，不能依赖前端隐藏按钮。

### 配置初始化

`ConfigStorageService` 增加重置操作：

1. 确保目标配置目录存在。
2. 从 classpath 读取 `config.yml`。该资源由 `krl-core` 模块随应用一同打包。
3. 在目标配置文件所在目录写入临时文件。
4. 优先使用原子移动将临时文件替换为目标 `Config.yml`；文件系统不支持原子移动时，回退为普通替换移动。
5. 返回磁盘配置路径及重置后的 UTF-8 文本。

采用替换写入而非先清空目标文件。最终结果仍是原内容被完整清除并替换为默认配置，但如果读取资源或写盘失败，原配置不会被破坏。临时文件在失败路径上必须清理。

`AnalysisController` 暴露 `POST /api/config/reset`。成功响应沿用现有配置查询响应结构：

```json
{
  "configPath": "/absolute/path/Config.yml",
  "content": "default yaml content"
}
```

### 程序退出

退出逻辑由独立服务负责，控制器只校验模式并调度退出：

1. `POST /api/runtime/shutdown` 首次调用时返回接受退出的响应。
2. 服务使用原子状态保证退出调度幂等，重复请求不创建第二条退出线程。
3. 独立的非守护线程短暂等待，让 HTTP 响应完成写回。
4. 线程关闭 `ConfigurableApplicationContext`，触发 Spring 对 Web 容器、分析线程池和其他托管 Bean 的生命周期清理。
5. 上下文关闭完成后退出 JVM，确保没有非托管线程继续占用进程。

生产代码中的进程退出动作必须可替换，以便测试验证调度和关闭行为时不会终止测试 JVM。

## 前端设计

### 配置初始化入口

在 Config 弹窗底部操作区增加“初始化配置”按钮，放在“从磁盘重新加载”按钮之前，并使用警示样式。

交互流程：

1. 用户点击按钮。
2. 显示确认信息：“此操作将覆盖磁盘中的 Config.yml，是否继续？”
3. 用户取消时不发送请求、不改变页面状态。
4. 用户确认后调用 `POST /api/config/reset`。
5. 成功后使用响应内容同时更新配置路径、编辑框内容和 `uploadState.configContent`。
6. 清除上次成功任务 ID 与输入签名，使后续分析使用新配置。
7. 更新操作按钮状态并提示初始化成功。
8. 请求失败时显示后端错误，不覆盖当前编辑内容。

### 程序退出入口

在顶部操作菜单的“下载 Excel”按钮之后增加“退出程序”按钮，并使用危险操作样式。该按钮只在桌面模式展示。

交互流程：

1. 用户点击按钮。
2. 显示确认信息，说明正在执行的分析也会被终止。
3. 用户确认后禁用页面操作并调用 `POST /api/runtime/shutdown`。
4. 接口接受请求后关闭操作菜单，页面显示“程序已退出，可以关闭此页面”。
5. 若请求在应用关闭前明确失败，则恢复可操作状态并显示错误。

配置初始化按钮同样只在桌面模式展示。`loadRuntimeStatus()` 获取运行模式后统一刷新这两个入口的可见性。

## 错误处理

- 内置 `config.yml` 缺失：返回 500，保留原磁盘配置与编辑框内容。
- 创建目录、写临时文件或替换目标失败：返回 500，尽力删除临时文件并保留原磁盘配置。
- 服务器模式调用任一接口：返回 403，不执行任何副作用。
- 重复退出请求：返回已接受或正在退出的成功语义，不重复调度。
- 前端只在收到配置重置成功响应后更新本地状态。

## 测试策略

### 配置服务测试

- 现有配置被内置默认配置完整覆盖。
- 重置返回的文本与磁盘内容一致。
- 默认资源读取或替换失败时不破坏旧配置。

### 控制器与退出服务测试

- 桌面模式允许配置重置，服务器模式返回 403。
- 桌面模式允许退出，服务器模式返回 403。
- 退出服务只调度一次。
- 测试替身能观测上下文关闭和进程退出顺序，且不终止测试 JVM。
- 现有 `/api/runtime/status` 与 `/api/config` 行为保持不变。

### 回归验证

- 运行 Maven 全量测试。
- 运行 Maven 打包，确认默认 `config.yml` 仍可从最终应用 classpath 读取。
- 在桌面模式手工验证按钮、确认提示、配置同步和进程退出。
- 在服务器模式手工验证两个按钮均隐藏，直接调用接口返回 403。

## 验收标准

- 桌面模式用户可以一键将磁盘配置恢复为项目内置默认配置。
- 初始化成功后，磁盘文件、编辑框与本次分析配置内容一致。
- 初始化失败不清空或损坏原配置。
- 桌面模式用户确认退出后，Java 进程自动结束并释放 Spring 托管资源。
- 服务器模式无法通过界面或直接 HTTP 请求使用上述两项能力。
- 新增测试、现有测试和项目打包均通过。
